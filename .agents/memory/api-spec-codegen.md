---
name: API spec is the source of truth for the generated client
description: Why a working server route can still crash the app with "useX is not a function", and the codegen workflow
---

The React Query hooks the smartboss app imports from `@workspace/api-client-react` are generated by **orval** from `lib/api-spec/openapi.yaml` (not from the Express routes). `lib/api-zod` is generated the same way.

**Rule:** When you add or change an endpoint in `artifacts/api-server/src/routes`, you MUST also add/update the matching operation (path + `operationId` + request/response schemas) in `lib/api-spec/openapi.yaml`, then run `pnpm --filter @workspace/api-spec run codegen`. The hook name is derived from `operationId` (e.g. `getWorkerDeleteRequests` → `useGetWorkerDeleteRequests`). Also reflect any new response fields in the relevant component `#/components/schemas` (e.g. `DeleteRequest`) or the typed data won't expose them.

**Why:** If the route exists on the server but not in the spec, orval never generates the hook. The component imports an undefined symbol and crashes at render with `TypeError: useX is not a function (it is undefined)` — looks like a client bug but the real gap is the spec. The server endpoint itself works fine via curl, which is misleading.

**How to apply:** On any "useX is not a function" crash for an API hook, check `openapi.yaml` has the operation; if missing, add it mirroring a sibling op and run codegen. Note: the `codegen` script also runs `typecheck:libs`, which currently fails on the unrelated `lib/integrations-gemini-ai` package (pre-existing `AbortError`/node-types errors) — orval generation still completes; verify the generated `api.ts` and typecheck just `lib/api-client-react` to confirm your change is clean.
